1. Where Our Skill Sits in Cargoes Logistics

A container terminal operates two core cargo flows — Import Discharge & Delivery and Export Receive & Loading. Each flow involves 10-15 operational steps managed by the Terminal Operating System (TOS). Our skill draws data from the Yard Planning, Gate Operations, and Operation Monitoring stages — the steps where dwell time accumulates and yard capacity is consumed.

6-Stage Analytics Pipeline

Algorithm Justification

Clustering: K-Means + Hierarchical

  • K-Means (primary): Multi-feature grouping on dwell_hours + teu_equivalent + storage_cost. Adapts to each terminal's patterns vs arbitrary thresholds.
  • Hierarchical (comparison): Ward linkage validates K-Means. Sampled to 10K for O(n²) memory.
  • Why not DBSCAN? Dwell data is continuous + right-skewed, not density-clustered.
Critical: Cluster on BEHAVIOR only. Categoricals dominate feature space → splits by type not dwell. Used for profiling AFTER assignment (RFM approach).

k-cap at 8: We test k=2–10 for complete charts, but only select from k≤8. At k=9–10, K-Means splits the same dwell band by 20ft vs 40ft, producing duplicate archetypes with identical recommendations. The silhouette gain is ~0.01 — not worth the interpretability loss. Full range shown in charts for transparency.

Forecasting: Holt-Winters + StatsModels

  • Holt-Winters (manual): Level + trend + 7-day seasonal. Auto-tuned α/β/γ parameters.
  • StatsModels HW: Library-optimized alternative. Both models compete; pipeline auto-selects the winner by MAPE.
  • 7-day seasonality: Weekday gate activity 30-40% higher than weekends.
  • Graceful fallback: If statsmodels unavailable → manual HW + SMA.

Reads Sections 1, 7, 8. Dollar figures, opportunity cost, actions. No jargon.

🏗️ Yard Planner

Reads Sections 3, 4, 5. Congested blocks, longest-dwelling types, clearance priorities.

📊 Analytics Team

Reads all sections. Silhouette, MAPE, elbow plot. Methodology verification.

Facts vs Estimates — What the Skill Realistically Delivers

The report deliberately separates two layers of output. Understanding this distinction is critical for how the skill should be used in practice:

100% factual — directly from the data

  • Container ZIMU4832917 has been in Block A01 for 127 days
  • 2,183 containers (4.5%) are sitting 34+ days
  • HMM and ZIM containers dwell 25–35% longer than average
  • Block R01 reefer plugs hit capacity every Friday
  • Vessel-to-rail containers dwell 2.1× longer than vessel-to-truck
  • Total storage charges collected: $8.8M over 6 months

These drive immediate action — no assumptions needed.

Estimated — depends on terminal demand

  • $11.5M in blocked throughput capacity (theoretical maximum)
  • $4.6M–$8.1M realizable at 40–70% demand fill rate
  • "Reducing dwell by 2 days frees 700 TEU slots"
  • "Each freed slot could earn $85/TEU in throughput"

These justify investment decisions — reported with explicit caveats and sensitivity ranges.

Key design decision: The actions in the report are valid regardless of the revenue estimate accuracy. "Clear container ZIMU4832917" is the right action whether the freed slot earns $85 or $0 — the container is still causing reshuffles, consuming yard management attention, and blocking crane paths. "Call ZIM about their slow pickups" is valid because the dwell comparison is a fact, not an estimate. The dollar figures serve a different purpose: they give the terminal manager a business case to justify hiring a customs liaison, adding a rail service, or building overflow capacity. The report separates these layers explicitly — the Priority Action List (Section 9) is the "pick up the phone" list for the yard planner; the Revenue & Cost Impact (Section 7) is the CFO's investment justification with honest 40–70% realization caveats.

Design Principles

📗 Insight Boxes + Faceted Charts

Green boxes answer: What am I looking at? Is this normal? What should I do? Box plots replaced with faceted histograms and horizontal bars — readable by non-analysts. Cumulative departure curve: "90% leave by day X."

🏷️ Segment Narrative Cards

Each cluster → WHO (dominant type, movement, flow) → BEHAVIOR (dwell, cost, overstay%) → WHY (root cause) → ACTION (recommendation). Labels differentiate 20ft vs 40ft when segments overlap.

💰 Revenue Transparency

Full tariff table (every rate sourced). Opportunity cost with caveat: "40-70% of theoretical max is achievable." What-if at ±30%, ±50%.

📉 Graceful Degradation

8 required columns → full report. Missing config → section warns instead of crashing. Progressive enrichment as optional data arrives.

Pluggable Configuration Design

The skill is designed to work with any terminal's data — not just our test dataset. Two optional JSON config files make the skill adaptable without code changes:

yard_config.json

Defines the terminal's physical layout: block IDs, TEU capacity per block, reefer plug counts, accepted container types, gate lane counts. When new data arrives: the user provides their terminal's block layout. Utilization percentages, congestion detection, and reefer plug analysis all adapt automatically. If omitted, the pipeline skips block-level analysis and notes the absence.

tariff_config.json

Defines the terminal's pricing: free days per movement type, tiered demurrage rates ($X/TEU/day at escalating thresholds), reefer and hazmat surcharges, throughput revenue per TEU. When new data arrives: the user supplies their rate card. Storage cost, opportunity cost, and what-if sensitivity all recalculate. If omitted, revenue section displays a warning — no crash.

4. What Failed, What We Fixed & Testing

Building this skill was iterative. Several design choices that seemed correct initially produced broken or misleading results. Each failure taught us something about the gap between textbook approaches and real terminal operations.

Problem 1: Synthetic Data Realism

What Failed

Random seeding with numpy produced statistically valid but operationally absurd data: 25% empty containers (real: 12–14%), block utilizations of 145–196% (physically impossible), overstay rates of 30%+ (real: 10–15%), and flat shipping line dwell times (7.3–8.0 days for every carrier). The data passed all schema checks but wouldn’t fool anyone who’s worked on a terminal.

How We Fixed It

Replaced flat randomisation with domain-calibrated cluster weights (15% fast / 52% normal / 22% extended / 8% overstay / 3% chronic), shipping line dwell multipliers (Maersk 0.85× vs ZIM 1.35×), flow-path multipliers (rail 1.4× vs truck 1.0×), and scaled block capacities to match a real terminal layout. Validated against live operational reports — distributions now match.

Problem 2: Hierarchical Clustering at 50K Scale

What Failed

AgglomerativeClustering with Ward linkage computes a full pairwise distance matrix — for 48K containers that’s 48K × 48K = 2.3 billion distance calculations. On a laptop, the script either hung for 30+ minutes or crashed with an out-of-memory error. The silhouette_score validation had the same O(n²) problem.

How We Fixed It

Hierarchical clustering now samples 10K rows via stratified random sampling, then assigns the full dataset using NearestCentroid. Silhouette score is also computed on a 10K sample. K-Means runs on the full dataset (it scales linearly). Result: pipeline completes in under 5 minutes on a laptop instead of hanging indefinitely.

Problem 3: K-Means Splitting by Category, Not Dwell Behavior

What Failed

With one-hot encoded categoricals (movement_type, container_type) in the feature set, K-Means produced 6 out of 7 segments all labelled “Standard Dwellers” with nearly identical dwell times (4.1–5.2 days). The algorithm was splitting by “is it import or export?” and “is it dry or reefer?” — not by dwell behavior. The recommendations section repeated the same advice 6 times.

How We Fixed It

Removed all categoricals from the clustering feature set. Cluster on dwell_hours + teu_equivalent + storage_cost only (the RFM approach — cluster on behavioral metrics, profile with demographics after). Segments now have distinct dwell ranges (0.5d to 10.6d) with unique labels. Added WHO / BEHAVIOR / WHY / ACTION narrative cards per segment with enriched profiles (dominant type, movement, flow path, overstay %).

Problem 4: Charts Not Stakeholder-Readable

What Failed

Box plots showed hundreds of outlier dots that looked like errors. Overlaid histograms had bars hiding behind each other. The yard heatmap x-axis showed raw datetime with nanosecond precision (2025-01-01T00:00:00.000000000). Technically correct, operationally useless for a yard planner.

How We Fixed It

Replaced box plots with faceted histograms (one panel per movement type) and horizontal bar charts with inline stats. Added a cumulative departure curve (“90% leave by day X”). Heatmap aggregated to weekly averages with clean date labels. Added insight boxes below every chart explaining what it means in plain business language.

Problem 5: Duplicate Segment Labels

What Failed

K-Means with k=7 produced multiple segments in the same archetype band — three clusters all labelled “Extended Stay” with identical recommendations repeated three times. No differentiation, no actionable insight.

How We Fixed It

Built a two-pass labelling system: assign base archetype by dwell range, then detect duplicates and differentiate by TEU size (20ft vs 40ft) and relative dwell position. Result: “Extended Stay (40ft, longest dwell)” vs “Extended Stay (20ft)” — each with unique, actionable identity and distinct capacity impact notes.

Future Enhancements

Near-term: BAPLIE integration for predictive yard planning, STORAGE movement type for warehousing containers (34–46d dwell), FULL/EMPTY split within movement types. Long-term: Equipment utilization correlation (RTG reshuffles increase with high-dwell blocks), real-time MCP connection to the Terminal Operating System for live alerts, and Port Community System integration for customs clearance tracking.

In summary: This skill sits at the intersection of Yard Planning, Gate Operations, and Operation Monitoring in the Cargoes Logistics chain. It answers the terminal operator's core questions — where is my yard congested, which containers should I prioritize for clearance, and what is the financial impact? Every problem we encountered — from unrealistic synthetic data to unusable clustering to unreadable charts — was rooted in a gap between what works in a textbook and what works in a real terminal. The fixes reflect operational reality, not statistical convenience.
